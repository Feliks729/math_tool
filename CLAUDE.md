# Math Tool — Математический инструмент

## Что мы строим

Образовательное приложение по математике — бесплатное, кроссплатформенное, для всех. Не заменитель школы, а сильный помощник. Охватывает 1–11 класс и ЕГЭ. Работает оффлайн, без регистрации, без платных функций.

Ядро приложения — **математический инструмент**. Всё остальное (теория, уроки, задания, интерактивы) строится вокруг него и использует его. Инструмент разрабатывается первым.

Два ориентира по духу:
- **GeoGebra** — по математической мощности
- **Duolingo** — по удобству, интерактивности и вовлечённости

Но не копия ни того ни другого. GeoGebra слишком сложен для школьника. Duolingo не про математику. Мы берём лучшее от обоих и делаем своё.

---

## Три настоящих принципа

Всё что делается в этом проекте проверяется по трём вопросам:

**1. Это понятно школьнику без объяснений?**
Интерфейс, инструменты, решения, подсказки — всё должно быть очевидным с первого взгляда. Если нужна инструкция — что-то сделано неправильно.

**2. Архитектура позволяет добавить это без переделки ядра?**
Каждый новый режим, каждая новая функция, каждый новый тип задания должны подключаться как модуль — не ломая то что уже работает.

**3. Решение выглядит как в школьном учебнике?**
Один движок объяснений используется везде — в заданиях, в свободном режиме, в демонстрациях. Стиль, обозначения, последовательность шагов — всегда такие, к каким ученик привык.

## Что инструмент умеет

Инструмент — единая интерактивная среда на сетке-клетке. Работает в двух контекстах:

**Свободный режим** — полная мощность, пользователь исследует сам.

**Режим задания** — инструмент встроен в урок или задание. Программа даёт условие, пользователь работает в том же инструменте, программа проверяет. Решение и объяснение появляются после попытки.

Внутри инструмента — несколько режимов работы (сколько нужно чтобы покрыть всю школьную математику и ЕГЭ, точное число не фиксировано). Каждый режим — отдельный модуль с единым контрактом входа/выхода.

**Семантическое поле** — ключевая идея. Поле не холст для рисования. Программа понимает математический смысл всего что на нём построено. Три точки соединённые отрезками — это треугольник со сторонами, углами и площадью. Кривая — это функция со свойствами. Программа всегда знает что на поле и может с этим работать.

**Двунаправленность** — формула идёт в график и график идёт в формулу. Оба направления работают.

**Единая клавиатура** — один математический компонент ввода, встроенный везде.

**Единая сетка** — одинаковая везде, с осями или без, всегда клетка как в тетради.

## Архитектурные решения — зафиксированные

Это решения которые не меняются в процессе без явной причины. Каждое из них принято осознанно.

---

### Решение 1 — CustomPainter для всей графики

Мы не используем `fl_chart`, `graphic` или другие готовые библиотеки для рисования поля, сетки, фигур и графиков. Всё рисуется вручную через `CustomPainter`.

**Почему:** только так можно сделать семантическое поле. Готовые библиотеки рисуют картинку — мы строим математическую среду. Плюс полный контроль над каждым пикселем: засечки параллельности, дуги углов, закрашенные области, умные аннотации — это невозможно в готовых решениях.

---

### Решение 2 — Riverpod для состояния

Не Provider, не GetX, не BLoC. Riverpod.

**Почему:** у нас сложное состояние которое читается из многих мест — активный инструмент, список объектов на поле, история undo/redo, текущий режим. Riverpod справляется с этим чисто, без лишних связей между компонентами. Легко тестируется.

---

### Решение 3 — Один движок объяснений

Не два стиля объяснений для разных контекстов. Один модуль `explanation_engine` который вызывается из любого места.

**Почему:** решение в свободном режиме и решение в задании должны выглядеть одинаково — как в школьном учебнике. Пользователь видит один и тот же стиль везде. Логика обслуживается в одном месте — легче поддерживать и улучшать.

---

### Решение 4 — Контракт режима

Каждый режим работы (геометрия, функции, числовая прямая, и любые будущие) реализует единый контракт:

```dart
abstract class MathMode {
  // Что принимает режим на входе (от урока или задания)
  void loadTask(MathTask task);
  
  // Что режим отдаёт наружу (результат работы пользователя)
  MathResult getResult();
  
  // Сброс состояния (новая задача или свободный режим)
  void reset();
}
```

**Почему:** добавить новый режим = создать новый класс который реализует этот контракт. Ядро не трогается. Уроки и задания общаются с любым режимом через один и тот же интерфейс — не нужно знать что внутри.

---

### Решение 5 — Паттерн Command для undo/redo

Каждое действие пользователя на поле — это объект команды. Стек команд = история действий.

```dart
abstract class FieldCommand {
  void execute();  // сделать
  void undo();     // отменить
}
```

**Почему:** undo/redo нужен везде и для всего. Паттерн Command даёт это автоматически для любого нового инструмента — просто реализуй execute и undo.

---

### Решение 6 — Слоистая архитектура поля

Поле состоит из четырёх слоёв, каждый рисуется отдельно в строгом порядке снизу вверх:

```
Слой 0 — Сетка          (фон, всегда виден)
Слой 1 — Объекты        (фигуры, графики, точки)
Слой 2 — Аннотации      (подписи, размеры, углы, засечки)
Слой 3 — Интерактив     (выделение, перетаскивание, превью)
```

**Почему:** каждый слой можно перерисовать независимо. Если пользователь двигает точку — перерисовывается только слой объектов и аннотаций, не вся сцена. Плюс слои можно скрывать по отдельности (например убрать аннотации для чистоты).

---

### Решение 7 — Числовая эквивалентность для проверки ответов

Проверка что `2(x+1)` и `2x+2` — это одно и то же делается не символьной алгеброй (сложно), а числовой подстановкой: подставляем несколько значений x, сравниваем результаты.

```dart
// Подставляем x = 1, 2, 3, -1, 0.5, π
// Если все результаты совпадают с точностью до 1e-10 — выражения эквивалентны
```

**Почему:** просто, надёжно, работает для 99% школьных случаев. Символьная алгебра — отдельная наука, нам не нужна.

## Как устроена работа

Есть два участника:

**Координатор (человек)** — даёт задачи, оценивает результат, одобряет или отклоняет решения, задаёт направление. Не пишет код, не принимает технические решения — это не его работа.

**Исполнитель (Claude)** — принимает технические решения, предлагает архитектуру, пишет код, объясняет каждый шаг так чтобы было понятно человеку без опыта в программировании. Если что-то непонятно в задаче — спрашивает. Если видит проблему — говорит сразу.

---

## Принцип одного шага

Каждое сообщение = одно конкретное действие. Не два, не три. Одно.

После каждого шага координатор пишет одно из трёх:

```
✅ готово          — шаг выполнен, всё работает
❌ ошибка: [текст] — что-то пошло не так
❓ вопрос: [текст] — что-то непонятно
```

Следующий шаг даётся только после ответа. Если пришла ошибка — сначала решаем её, потом идём дальше.

## Правила кода

**Шапка в каждом файле — обязательно:**

```dart
// ============================================
// ФАЙЛ: lib/core/math_model/math_point.dart
// ЧАСТЬ ПЛАНА: 1.2.1 — MathPoint
// ЧТО ДЕЛАЕТ: базовый объект точки на поле
// ЗАВИСИТ ОТ: constants.dart
// ============================================
```

**Комментарии на русском, по смысловым блокам:**

```dart
// Вычисляем расстояние между двумя точками по теореме Пифагора
double distance = sqrt(pow(b.x - a.x, 2) + pow(b.y - a.y, 2));
```

**Код всегда полный.** Никаких `// остальное то же самое`. Вставляется весь файл целиком или точно указанный фрагмент с адресом куда вставить.

**Имена файлов — строчными через подчёркивание:**
```
math_point.dart      ✅
MathPoint.dart       ❌
```

**Имена классов — CamelCase:**
```dart
class MathPoint       ✅
class math_point      ❌
```

**Импорты — строго в таком порядке:**
```dart
// 1. Flutter / Dart
import 'package:flutter/material.dart';
// 2. Внешние пакеты
import 'package:riverpod/riverpod.dart';
// 3. Наши файлы — от общего к частному
import '../../core/constants.dart';
import '../grid_settings.dart';
```

## Правила плана

**План не меняется молча.** Если нужно изменить — фиксируем явно:

```
ИЗМЕНЕНИЕ ПЛАНА
Пункт: [номер]
Было: [старая версия]
Стало: [новая версия]
Причина: [почему]
```

**Порядок строгий.** Нельзя перейти к части 3 пока не готова часть 2. Исключение — только если это решено и записано.

**После каждого завершённого раздела — чеклист проверки.** Новый раздел начинается только когда все пункты чеклиста отмечены.

## Правила инструментов (Code / Chat)

Это не фиксированные правила — решается в моменте по ситуации. При каждом шаге Claude указывает:

```
🛠 ГДЕ ДЕЛАЕМ: Code / Chat
⚡ УСИЛИЕ: low / medium / high / max
```

**Общая логика:**
- Создать файл по понятной структуре → Code, medium
- Написать сложную математическую логику → Code, high
- Архитектурное решение которое сложно переделать → Code, max
- Обсудить идею, написать текст, спланировать → Chat
